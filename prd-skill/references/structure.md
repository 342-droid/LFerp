## 1. 文档信息

**目的**：让读者快速了解文档基本情况，便于版本管理和追踪。

**必填字段**：
- 文档标题
- 产品/功能名称
- 版本号（格式：v1.0, v1.1）
- 作者
- 最后更新日期
- 状态（草稿 / 评审中 / 已确认 / 已归档）
- 审阅人/相关干系人

**可选字段**：
- 变更记录表（版本 / 日期 / 修改人 / 变更说明）

## 2. 背景与目标

**目的**：回答「为什么做这个」，帮助读者理解决策上下文。

### 2.1 背景

说明以下内容：
- **现状问题**：当前存在什么痛点、机会或压力
- **触发因素**：为什么现在做（业务目标、数据驱动、竞品压力、用户反馈等）
- **历史上下文**：如有相关历史决策或前序版本，简要说明

写法要点：
- 用数据支撑问题陈述（如有）：「目前注册转化率仅 23%，行业均值 40%」
- 避免直接跳到解决方案

### 2.2 目标

区分两类目标：
- **业务目标**：对公司/产品的价值（提升留存、增加收入、降低成本等）
- **产品目标**：本次迭代具体要达成的结果

每个目标尽量量化：
- ✅「将注册转化率从 23% 提升至 35%」
- ❌「提升注册转化率」

### 2.3 范围说明（In Scope / Out of Scope）

明确列出：
- **本期包含**：哪些功能/场景在这次迭代内
- **本期不包含**：哪些相关功能有意不做（避免歧义）

## 3. 用户分析

**目的**：说清楚「为谁做」，建立共同的用户心智模型。

### 3.1 目标用户

描述核心用户群体：
- 人口/角色特征（年龄、职业、技术熟练度等）
- 使用场景和动机
- 当前行为方式（现在怎么解决这个问题）

### 3.2 用户痛点

列出 3-5 个具体痛点，与需求形成对应关系。

格式建议：
```
痛点 1：[简短标题]
描述：[1-2句具体说明]
影响：[频率/严重程度]
```

### 3.3 用户旅程（可选，适用于复杂功能）

用流程图或分阶段描述用户完成核心任务的全程，标出关键节点和情绪低谷点。

## 4. 功能需求

**目的**：详细描述产品要做什么，是 PRD 的核心章节。

### 结构建议

按功能模块分组，每个模块包含：
- **模块名称与说明**
- **功能列表**（带优先级）
- **详细需求描述**
- **验收标准**
- **异常/边界情况**

### 优先级标注

推荐使用以下优先级体系：

| 标记 | 含义 | 说明 |
|------|------|------|
| P0 | 必须有 | 缺少即无法上线，阻断核心流程 |
| P1 | 重要 | 显著影响用户体验，首期强烈建议 |
| P2 | 一般 | 有则更好，可延后迭代 |
| P3 | 低优先级 | 锦上添花，酌情排期 |

### 功能需求编号

格式：`FR-[模块缩写]-[序号]`
例：`FR-REG-001`（注册模块第1条需求）

### 每条需求的写法

```
FR-XXX-001 [需求标题]
优先级：P0
描述：[详细说明用户能做什么/系统会做什么]
验收标准：
  - [ ] [可测量的验收条件1]
  - [ ] [可测量的验收条件2]
异常情况：[边界条件和错误处理]
```

## 5. 非功能需求

**目的**：定义产品质量属性，通常由研发、测试、运维关注。

### 常见非功能需求类别

**性能**
- 页面加载时间（如：首屏 < 2s，TP95）
- 接口响应时间（如：核心接口 < 200ms）
- 并发支持（如：支持 10,000 QPS）

**可用性**
- 系统可用性目标（如：99.9% SLA）
- 故障恢复时间（RTO / RPO）

**安全性**
- 数据加密要求
- 权限控制模型
- 合规要求（GDPR、等保等）

**兼容性**
- 浏览器兼容（如：Chrome 90+, Safari 14+）
- 设备/分辨率支持
- 操作系统版本

**可扩展性**
- 数据量增长预期
- 未来功能扩展预留

## 6. 方案设计（可选）

**何时包含此章节**：当 PRD 面向研发团队，或需要描述 UI/交互/架构时。

### 6.1 信息架构 / 页面结构

说明新功能在产品中的位置和导航层级。

### 6.2 核心交互流程

用流程图或步骤说明用户操作主路径和关键分支。

格式建议：
```
步骤1：[用户操作] → 步骤2：[系统响应] → 步骤3：[用户操作]
```

### 6.3 UI 说明 / 原型引用

- 链接到原型（Figma / Axure / 截图）
- 关键界面的字段说明
- 空状态、加载状态、错误状态的处理

### 6.4 数据流 / 接口概述（技术 PRD）

- 关键接口的输入输出说明
- 数据来源和存储位置
- 与第三方服务的集成

## 7. 数据指标

**目的**：定义如何衡量产品是否成功。

### 北极星指标

定义 1 个最核心的成功指标。

### 过程指标

5-8 个追踪性指标，覆盖以下维度：
- **获取**：新用户/新功能使用率
- **激活**：完成核心操作的用户比例
- **留存**：回访/复用率
- **变现**（如适用）：转化率、ARPU
- **推荐**（如适用）：NPS、分享率

### 反指标（Guardrail Metrics）

定义「不能因此变差」的指标，防止优化一个指标伤害另一个。

### 数据采集要求

说明需要埋点的事件列表（如需新增埋点）。

## 8. 里程碑计划

**目的**：明确时间预期，便于资源对齐和进度追踪。

### 推荐格式

| 里程碑 | 交付内容 | 目标日期 | 负责人 |
|--------|----------|----------|--------|
| M1 设计评审 | UI 原型确认、需求冻结 | YYYY-MM-DD | PM |
| M2 研发完成 | 功能开发完毕、自测通过 | YYYY-MM-DD | Tech Lead |
| M3 测试完成 | QA 测试通过、Bug 修复 | YYYY-MM-DD | QA |
| M4 灰度上线 | 5% 用户灰度、监控观察 | YYYY-MM-DD | PM |
| M5 全量发布 | 100% 用户可用 | YYYY-MM-DD | PM |

## 9. 风险与依赖

**目的**：提前识别潜在阻碍，制定应对方案。

### 风险矩阵

| 风险 | 概率 | 影响 | 应对方案 |
|------|------|------|----------|
| [风险描述] | 高/中/低 | 高/中/低 | [降低风险或应急方案] |

### 外部依赖

列出需要其他团队/系统支持的事项：
- 依赖方
- 需要的支持内容
- 预期完成时间
- 当前状态

## 10. 附录（可选）
