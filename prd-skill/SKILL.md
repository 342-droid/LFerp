# PRD 产品需求文档技能

## 概述

本技能指导 Claude 生成专业、结构完整的 PRD 文档，适用于从早期概念到详细规格的各阶段产品需求场景。

## 快速参考

| 任务 | 方法 |
|------|------|
| 从零创建完整 PRD | 按标准模板填写所有章节，详见下方 |
| 仅写某个功能需求 | 使用「功能规格模块」，见 references/modules.md |
| 优化/重写现有 PRD | 先阅读原文，识别缺失章节，按模板补全 |
| 用户故事拆解 | 使用「用户故事模块」，见 references/modules.md |

## 工作流程

### 第一步：收集关键信息

在开始写 PRD 前，先确认以下信息（若用户已提供可跳过）：

1. **产品/功能名称** — 这是什么？
2. **目标用户** — 谁在用？解决谁的问题？
3. **核心问题** — 现在有什么痛点或机会？
4. **预期目标** — 成功的定义是什么（指标/里程碑）？
5. **范围边界** — 这期做什么？不做什么？
6. **文档用途** — 给谁看？（研发评审 / 融资 / 内部对齐）

若信息不全，用一次提问收集最关键的缺口，不要逐条追问。

### 第二步：选择文档深度

根据用户意图和已有信息选择合适深度：

| 深度 | 适用场景 | 篇幅参考 |
|------|----------|----------|
| **精简版** | 快速对齐、内部讨论、早期探索 | 500-1000 字 |
| **标准版** | 常规功能开发、迭代需求 | 1500-3000 字 |
| **完整版** | 核心产品、大型功能、跨团队协作 | 3000-6000 字 |

默认使用**标准版**，除非用户明确指定或内容复杂度明显超出。

### 第三步：撰写文档

按照下方标准结构输出 PRD。若生成的是文件（docx/md），参照 docx/pdf 技能的最佳实践。

---

## 标准 PRD 结构

```
1. 文档信息
2. 背景与目标
3. 用户分析
4. 功能需求
5. 非功能需求
6. 方案设计（可选）
7. 数据指标
8. 里程碑计划
9. 风险与依赖
10. 附录（可选）
```

详细章节说明见 `references/structure.md`。
各章节写作模板和示例见 `references/templates.md`。
常用模块（用户故事、功能矩阵、竞品对比）见 `references/modules.md`。

## AI 可识别的 PRD 格式

阅读或优化已有 PRD 时，按以下约定识别文档结构，便于补全、改写或校验。

### 整体结构

- 按 **10 章标准结构** 解析；章节可用「一、二、三」或「1. 2. 3.」编号，子节用 `### 1.1`、`### 2.1` 等。
- 缺失的章节可识别并建议补全。

### 编号与命名

| 类型 | 格式 | 示例 |
|------|------|------|
| 功能需求 | `FR-[模块缩写]-[序号]` | `FR-REG-001`、`FR-SEARCH-001` |
| 非功能需求 | `NFR-001` | `NFR-001` |
| 用户故事 | `作为 [角色]，我希望 [功能]，以便 [价值]` | 技能定义句式 |

### Markdown 与表格

- **标题**：`#` / `##` / `###` 对应文档 / 章 / 节。
- **表格**：文档信息、变更记录、功能列表、里程碑、风险、指标、埋点等。
- **验收标准**：`- [ ]` 复选框列表。
- **优先级**：`P0` / `P1` / `P2` / `P3` 或 Must / Should / Could。

### 文档深度

| 深度 | 篇幅 | 可识别特征 |
|------|------|------------|
| 精简版 | 约 500–1000 字 | 章节少、描述简略 |
| 标准版 | 约 1500–3000 字 | 10 章基本齐全、有表格与验收标准 |
| 完整版 | 约 3000–6000 字 | 含方案设计、附录、多表格 |

### 可复用模块

识别并保持与 `references/modules.md` 一致的格式：用户故事卡片、功能优先级矩阵、竞品分析表、权限矩阵、状态流转、通知规则、埋点表、FAQ/设计决策、术语表、相关文档链接。

### 文档信息与元数据

- 版本（如 `v1.0`）、状态（草稿/评审中/已确认/已归档）、日期（`YYYY-MM-DD`）、变更记录表。

### 特殊扩展

- **技术 PRD**：出现接口设计、数据模型、API、TP99/QPS 等 → 按技术 PRD 补全/优化。
- **融资/对外 PRD**：出现市场规模、竞品、商业模式、团队等 → 按对外 PRD 侧重优化。
- **待确认**：文中 `[待确认: xxx]` 视为信息缺口，可生成待确认清单或后续补全。

## 核心写作原则

### ✅ 好的需求描述

- **具体可测量**：「用户可在 3 步内完成注册」而非「注册流程要简单」
- **以用户为中心**：说明功能解决什么问题，而非只描述功能本身
- **明确优先级**：用 P0/P1/P2 或 Must/Should/Could 标注
- **区分需求与方案**：需求说「用户能看到历史订单」，方案说「用列表展示」
- **可验收**：每个需求都能写出对应的验收标准

### ❌ 避免的写法

- 模糊表达：「系统要快」「界面要美观」「用户体验要好」
- 研发语言：「用 Redis 缓存」「接口返回 200」（除非是技术 PRD）
- 遗漏边界：不说清楚异常流程和降级方案
- 没有优先级：所有需求平铺，无法指导排期

## 枚举值处理规则（重要）

### 严格遵循资料中提供的枚举

当用户提供的资料（需求文档、接口文档、原型说明等）中已明确列出枚举值时，**必须严格按照资料中的枚举值输出，不得自行增减**。

**正确做法：**
- 资料说状态为「待审核、已通过、已拒绝」→ PRD 只写这三个状态，不额外补充「已撤销」「已过期」等
- 资料说类型为「普通用户、VIP 用户」→ 只列这两类，不自行扩展

**禁止做法：**
- 在资料已有枚举基础上，自行补充「可能有用」的枚举值
- 认为「这个场景一定还需要 XX 状态」就直接加入文档

### 认为枚举可能不完整时的处理方式

如果 AI 判断资料提供的枚举**可能存在遗漏，且该遗漏对需求完整性影响较大**，必须：

1. **先完成文档**，严格按资料中的枚举值输出
2. **在文档末尾或完成后**，以醒目方式提示用户，例如：

> ⚠️ **枚举补充建议（需您确认）**
> 
> 我注意到「XX 状态」枚举目前包含：A、B、C。
> 以下枚举值可能需要补充，请确认是否加入：
> - **D（说明原因）**：……
> - **E（说明原因）**：……
> 
> 如需补充，请告知，我将更新文档。

3. **未经用户明确同意，不得将建议的枚举值写入正式 PRD 内容**

## 输出格式规范

### 当生成 Markdown 文档时

- 使用 `#` `##` `###` 层级标题
- 表格用于功能列表、优先级矩阵、竞品对比
- 需求编号格式：`FR-001`（功能需求）、`NFR-001`（非功能需求）
- 用户故事格式：`作为 [角色]，我希望 [功能]，以便 [价值]`

## 特殊场景处理

### 用户只给了一个模糊想法

先基于现有信息写出 PRD 草稿，在每个信息缺口处用 `[待确认: xxx]` 标注，最后列出需要用户补充的关键问题。

### 用户要优化已有 PRD

1. 阅读原文，识别结构缺失、逻辑漏洞、描述模糊的地方
2. 输出改进建议（可用批注/对照方式）
3. 提供重写版本

### 用户要写技术 PRD（面向研发）

增加以下章节：
- 接口设计（API 契约）
- 数据模型变更
- 技术约束与假设
- 性能指标（TP99、QPS 等）

### 用户要写融资 / 对外 PRD

重点强调：
- 市场规模与机会
- 竞品分析与差异化
- 商业模式与变现路径
- 团队与执行能力
